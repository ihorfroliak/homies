# 04 — Фінансова архітектура (Stripe Connect) і регуляторна стратегія

## 1. Фінансова архітектура після рішення D2

### 1.1 Потік грошей (цільовий)

```
Guest оплата (card/BLIK/Apple Pay)
  → Stripe PaymentIntent (SCA/3DS автоматично)
  → Connect: destination charge на connected account host-а
     з application_fee (managed fee / комісія)
  → Stripe тримає кошти (регуляторно чисто, KNF-ліцензія не потрібна)
  → Transfer/Payout host-у за розкладом (затримка = наше захисне вікно)
  → Клінінг-фі: окремий transfer на connected account HeyHomie
     (спліт-механіка з коробки)
Refund/суперечка → Stripe refund/reversal → ledger-дзеркало оновлюється
```

### 1.2 Що лишається нашим (Ledger-дзеркало)

Stripe — джерело правди по грошах; **наш ledger — джерело правди по
бізнес-сенсу** грошей: прив'язка до бронювань/суперечок/штрафів,
CM-аналітика, звірка (щоденний reconciliation Stripe Balance ↔ ledger,
нульова толерантність), нарахування, які Stripe не знає (внутрішні
штрафи, компенсації купонами).

### 1.3 Закриті питання Part-2 ревю

| Проблема (ревю) | Рішення через Connect |
|---|---|
| Ліцензія платіжної установи | не потрібна: кошти у Stripe |
| Escrow-механіка | керована затримка payout-ів |
| KYC host-ів | Connect onboarding (identity+IBAN+compliance) |
| Refund після payout | negative balance connected account → Stripe стягує з наступних виплат; наш ліміт експозиції зверху |
| Chargeback-и | Stripe Disputes API + наш процес доказів |
| SCA/3DS, Apple/Google Pay, BLIK (критично для PL!) | з коробки |
| Санкційний скринінг виплат | Stripe compliance |
| FX | сlearing у валюті платежу; правило: refund = валюта і сума оригіналу |
| Спліт клінінг-фі HeyHomie | multi-party transfers |

### 1.4 Що Stripe НЕ закриває (наша робота)

VAT-логіка комісії (23% на managed fee — інвойси host-ам), міський
туристичний збір (конфіг ринку), DAC7-звіт (Stripe дає дані, звітуємо
ми), внутрішні штрафи/компенсації, резервний фонд гарантій (V2,
% від GMV на рахунку компанії), фінмодель CM по об'єктах.

### 1.5 Managed-специфіка

Договір з host-ом = агентський (діємо від його імені) — податково
критична розвилка (агент vs суборенда: суборенда = наш VAT на всю
виручку і наш ризик порожніх ночей). **Рада: агентська модель на
старті**; суборенда (гарантований дохід host-у) — лише для топ-об'єктів
на даних року+ (це продукт «Homies Guarantee», V2/V3).

## 2. Регуляторна стратегія (legally scalable operating model)

### 2.1 Реєстр вимог (Польща, старт)

| Домен | Вимога | Дія |
|---|---|---|
| PSD2/платежі | закрито Stripe Connect (D2) | договір, компліанс-анкети |
| Короткострокова оренда | PL: закон про usługi hotelarskie — повідомлення gminie, книга гостей для об'єктів; зміни законодавства STR у ЄС моніторити | поле реєстрації об'єкта; чек-лист онбордингу host-а; юр-підписка на зміни |
| EU STR Regulation (2024/1028, застосування з 2026) | обмін даними про бронювання з органами через single digital entry point | архітектурно: експорт-канал даних (закладено в бізнес-пакеті) |
| DAC7 | річна звітність доходів host-ів | збір TIN при онбордингу; звіт з ledger+Stripe |
| GDPR/RODO | зони PII (є в пакеті), consent-режим аналітики, DPIA на скоринг, retention schedule | до публічного запуску; DPA зі Stripe/хмарою |
| DSA | notice-and-action на контент, точка контакту | процедура скарг (є в Disputes/модерації — формалізувати) |
| Consumer protection ЄС | повна ціна у видачі (є), right of withdrawal — **виняток для дат розміщення** (ст. 16(l) 2011/83/EU) — політики скасування легальні, зафіксувати в T&C | юрист-рев'ю T&C |
| Туристичний збір | локальний конфіг ринку | Pricing-модуль |
| Agentська модель | договір Homies↔Host (управління) + regulamin платформи | юрист (разова інвестиція, шаблонизується) |
| Бухоблік PL | KSeF-інвойсинг (обов'язковий e-invoicing PL!) | бухгалтерський сервіс з KSeF-інтеграцією |

### 2.2 Принцип масштабування регуляторики

«Регуляторика як конфігурація»: кожен ринок = набір параметрів
(податки, реєстраційні поля, звіти, T&C-версія) поверх незмінного ядра.
Новий ринок відкривається чек-листом (юр-аналіз → конфіг → локальний
бухгалтер), без переписування системи. Це вже закладено в бізнес-пакеті
(markets) — рада підтверджує правильність.
