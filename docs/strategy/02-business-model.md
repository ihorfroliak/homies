## 2. Аудит функцій бізнес-пакета за бізнес-цінністю

Критерії: дохід / витрати↓ / retention / перевага / юридична
необхідність. Вердикти: **KEEP** (MVP), **SIMPLIFY**, **DEFER**, **CUT**.

| Функція (з docs/business) | Вердикт | Обґрунтування → метрика |
|---|---|---|
| Instant Book | KEEP | конверсія (Conversion Rate) — ядро |
| Request to Book | **DEFER V1** | ускладнює MVP; managed-об'єкти завжди instant → CR↑ |
| Політики скасування ×3 | **SIMPLIFY: 1 політика (Помірна) в MVP** | автоматизація refund-ів без матриці; CR не страждає |
| Зміни бронювання (BookingChange) | DEFER V1 | рідкісний кейс на малому обсязі; підтримка вручну |
| Double-blind відгуки | KEEP | Trust → Retention, NPS |
| Messaging + маскування | KEEP (юр.+ дезінтермедіація) | Take Rate protection |
| Disputes 4 рівні | **SIMPLIFY MVP: 2 рівні** (сторони → людина; апеляція = та сама людина+засновник) | на <100 бронювань/міс матриця рішень надлишкова; повна модель — V1 |
| Депозитна відповідальність | KEEP (спрощено: Stripe hold) | захищає supply → Host Retention |
| Loyalty | **DEFER V2+** (підтверджено радою повторно) | retention на малій базі дає CRM, не програма балів |
| CRM-модуль (supply-акквізиція) | **CUT як модуль** → таблиця+процес засновника | нуль доходу від софта тут; писати код — палити місяці |
| Marketing/атрибуція-модуль | DEFER V1 | до платного трафіку нема чого атрибутувати |
| Search-модуль (Meilisearch) | **SIMPLIFY MVP: Postgres-пошук** | <500 об'єктів не потребують окремого двигуна; Meili у V1 |
| Smart Pricing ML | DEFER V2, **але**: rule-based прайсинг managed-об'єктів — MVP | ADR/Occupancy — головні операторські метрики |
| KYC-провайдер | **SIMPLIFY: Stripe Connect onboarding = KYC host-ів** | D2 закриває; свій T&S-мінімум лише поверх |
| DAC7/податкова звітність | KEEP (юр. необхідність) | штрафи; частково зі Stripe-звітів |
| Автомодерація ML | DEFER V1; MVP = чек-лист людини | 50 оголошень модерує засновник; фокус на швидкість supply |
| Operations/ERP задачі | **KEEP і ПІДНЯТИ ПРІОРИТЕТ** | це ядро managed-моделі: клінінг-SLA = обіцянка бренду → NPS, CM |
| Channel manager (лістинг на Airbnb/Booking) | **ПІДНЯТИ: V1 → MVP-критично** для managed-об'єктів | чужий трафік = наша виручка з дня 1 → GMV, Occupancy |
| Admin Live Monitoring real-time | SIMPLIFY MVP: черги+щоденний дайджест | real-time до 100 бронювань/день — іграшка |
| Мобільні застосунки guest/host | DEFER (web-first) | CR mobile-web достатня; апки — retention-фаза |
| Мобільний застосунок Field Worker | **KEEP MVP** (є в HeyHomie-планах — переиспользувати!) | операційна досконалість = маржа |

**Найважливіша зміна пріоритетів:** Operations + Channel Manager
піднімаються в ядро MVP (це managed-модель), Search/CRM/Marketing/
повні Disputes — вниз. MVP-визначення змінилося з «петля довіри
маркетплейсу» на **«петля виручки оператора»**: об'єкт onboarded →
листується всюди → бронювання (свої+OTA) → клінінг за SLA →
виплата host-у → звіт. Це можна запустити з 10 об'єктами і рахувати
CM з першого місяця.

## 3. Юніт-економіка managed-об'єкта (каркас для фінмоделі, Чат 13)

```
Виручка об'єкта/міс     = ADR × ночей × occupancy
Дохід Homies/міс        = managed fee % × виручка + клінінг-маржа × turnover-ів
Змінні витрати          = клінінг-собівартість, витратні матеріали,
                          OTA-комісії (на чужі канали), платіжні комісії,
                          підтримка/інциденти
CM2/об'єкт              = дохід − змінні витрати
Поріг життєздатності    : CM2 > 0 при occupancy ~65%, ціль 75%+
Density-бонус           : клінінг-логістика дешевшає з к-стю об'єктів у районі
```
