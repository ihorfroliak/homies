# ADR-0006: Власна JWT-автентифікація для MVP; Keycloak — як опція

- **Статус:** Accepted
- **Дата:** 2026-06-25

## Контекст
Identity — це generic-піддомен (вирішена проблема). У реальному проді часто беруть готове (Auth0, Cognito, Keycloak). Для портфоліо хочеться показати розуміння теми, але для MVP-моноліту окремий сервіс Keycloak — це додатковий операційний наклад і ще один компонент у деплої.

## Рішення
Для MVP реалізуємо **власну JWT-автентифікацію** в модулі `auth` (менше рухомих частин, простіший деплой, нижчий burn). **Keycloak** документуємо як шлях апгрейду. Зовнішній контракт `auth` спроєктовано так, щоб перехід майже не змінював API: за потреби `login`/`refresh` делегуються OIDC-ендпойнтам Keycloak, а в нас лишаються `/me` і профіль.

## Наслідки
**Позитивні:** простий старт і повний контроль (корисно для навчання); легкий деплой; чесна історія на співбесіді («у проді взяв би Keycloak, своя реалізація — для демонстрації»).

**Негативні:** власна автентифікація — зона ризику безпеки, що вимагає ретельності (хешування Argon2id, ротація refresh-токенів, захист від перебору); функції «з коробки» (MFA, соціальний логін) доведеться писати самому або відкласти.

## Розглянуті альтернативи
**Keycloak одразу** — глибше й переносиміше, але +сервіс і +операційна робота. **Cognito** — швидко, але вендор-лок і слабша демонстрація глибини.
