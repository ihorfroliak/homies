# ADR-0004: Подієва інтеграція через outbox + легку шину; REST для синхронних читань

- **Статус:** Accepted
- **Дата:** 2026-06-25

## Контекст
Модулі мають реагувати на доменні події (бронь завершено → створити прибирання), а сервіс Notifications і платформа даних — це справжні асинхронні споживачі. Питання: як інтегрувати модулі й чи вводити Kafka одразу.

## Рішення
**Синхронно (REST або прямий виклик у моноліті)** — там, де відповідь потрібна негайно (напр. Booking запитує котирування в Pricing). **Асинхронно (події)** — для реакцій, що можуть статися «трохи згодом». Події публікуємо через **Postgres outbox-патерн**: дані й подію записуємо в одній транзакції, окремий процес потім надійно публікує її в **легку шину (NATS/Redis)**. Важку **Kafka/MSK** вводимо лише на етапі платформи даних (Фаза 2) для стрімінгу інжесту.

## Наслідки
**Позитивні:** надійна публікація без втрати подій (дані й подія атомарні); роз'єднаність модулів; легкий старт без важкої інфраструктури.

**Негативні:** eventual consistency — реакції й пошук відстають на секунди; складніший дебаг розподілених потоків; потрібен механізм дедуплікації на споживачі (за `eventId`).

## Розглянуті альтернативи
**Kafka з першого дня** — зайва складність і вартість для MVP. **Синхронні виклики на всі реакції** — жорстка зв'язаність і крихкість (падіння одного модуля валить ланцюг).
