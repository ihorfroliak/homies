# ADR-0001: Monorepo для всього проєкту

- **Статус:** Accepted
- **Дата:** 2026-06-25

## Контекст
Проєкт складається з багатьох частин: бекенд-модулі, веб-застосунки, мобільні застосунки, платформа даних, інфраструктура. Між ними є спільні контракти (OpenAPI, AsyncAPI), які треба тримати узгодженими. Постає питання: тримати все в одному репозиторії чи розбити на багато (polyrepo)?

## Рішення
Тримаємо весь проєкт в одному **monorepo** з чіткою структурою (`services/`, `apps/`, `data/`, `infra/`, `docs/`). Зміна контракту і всіх його споживачів робиться в одному pull request.

## Наслідки
**Позитивні:** атомарні зміни (контракт + код, що його використовує, в одному PR); єдина історія й нумерація версій; простіше для соло-розробника; охайна структура справляє враження в портфоліо.

**Негативні:** на масштабі CI ускладнюється (потрібні вибіркові збірки лише зміненого); репозиторій росте; права доступу — все або нічого.

## Розглянуті альтернативи
**Polyrepo** (окреме репо на кожен сервіс) — гнучкіші незалежні релізи й чіткіші межі команд, але значно важче синхронізувати контракти й непотрібний оверхед для однієї людини.
