# 04 — Архітектура модулів і даних

## 1. Архітектура модулів

Базуємось на карті контекстів Chat 01
([bounded-contexts.md](../diagrams/bounded-contexts.md)) і рішенні про
модульний моноліт (ADR-0001). Бізнес-аналіз пакета виявив **6 нових
контекстів** (обґрунтування в документі 07):

### 1.1 Повна карта модулів (17)

| Модуль | Відповідальність | Ключові сутності (власник даних) | Було/нове |
|---|---|---|---|
| Identity | акаунти, ролі, сесії, згоди, делегування (Co-Host) | User, RoleAssignment, Consent, Delegation | було |
| Listings | каталог об'єктів, гео, фото, правила проживання, health-score | Listing, Photo, HouseRules, Amenity | було |

| Payments | інтеграція з провайдерами, холди/capture, повернення, chargeback-и | Payment, Refund, PaymentMethod (токен) | було |
| **Ledger** | подвійний облік: escrow, баланси host-ів, комісії, штрафи, виплати | Account, JournalEntry, Payout | **нове** |

Deployment (ADR-0001 не змінюється): все — модулі моноліту, окремо лише
Notifications і ML-serving. Ledger — модуль з **найсуворішою** ізоляцією
(тільки власний API, жодного прямого запису з інших модулів).

### 1.2 Правила взаємодії модулів

1. Синхронно (виклик application-сервісу) — лише коли потрібна відповідь
   зараз: Booking → Pricing (Quote), Booking → Payments (charge).
2. Асинхронно (події) — все інше: реакції, проєкції, аналітика.
3. Модуль не читає чужих таблиць. Крапка. Читання даних іншого контексту —
   через його API або власну проєкцію з подій.
4. Адмін-функції — не окремий моноліт «адмінки»: кожен модуль виставляє
   свої admin-ендпоінти + read-моделі; Admin Center — окремий фронтенд
   над ними (BFF).

## 2. Архітектура даних

### 2.1 Три контури даних

| Контур | Сховище | Призначення | Латентність |
|---|---|---|---|
| **OLTP** | PostgreSQL (+PostGIS), схема на модуль (ADR-0003) | транзакційна робота продукту | мс |
| **Real-time проєкції** | Redis / материалізовані read-моделі, WebSocket-пуш | Live Monitoring адмінки, лічильники, черги | секунди |
| **Аналітичний** | події → S3 (lake) → DWH → dbt-марти → BI | KPI, когорти, unit-економіка, ML-фічі | хвилини-години |

### 2.2 Хто які дані створює і хто використовує

| Дані | Створює | Основні споживачі |
|---|---|---|
| Профілі, ролі, згоди | Identity | всі модулі (id-посилання), T&S |
| Оголошення, фото, гео | Host через Listings | Search, Booking, Moderation, ML |
| Календар, бронювання | Booking | Host/Guest апки, Operations, Ledger, DWH |
| Котирування, правила цін | Pricing | Booking, Host-аналітика, ML |
| Платежі, повернення | Payments | Ledger, T&S (фрод), Finance Ops |
| Проводки, баланси, виплати | Ledger | Host-фінанси, Finance Ops, DWH |
| Повідомлення | Messaging | сторони чату, T&S (автодетект), Disputes (докази) |
| Відгуки | Reviews | Search (ранжування), Host Quality Score |
| Ризик-скори, KYC | T&S | Payments (step-up), Admin черги |
| Суперечки, докази | Disputes | Ledger (компенсації), T&S, DWH |
| Тікети | Support | SLA-звіти, CSAT, продуктова аналітика |
| Події поведінки (перегляди, кліки) | фронтенди → event bus | Search-ранжування, рекомендації, Marketing |

### 2.3 Як дані потрапляють в Admin Center у real-time

1. Кожна доменна подія (ADR-0004) має консюмер-проєктор адмінки, який
   оновлює read-моделі черг і лічильників (нові суперечки, KYC-черга,
   заїзди сьогодні, платежі/хв).
2. Read-моделі пушаться у фронтенд адмінки через WebSocket/SSE.
3. Деталь кейса адмінка читає через admin-API відповідного модуля
   (з RBAC/ABAC-перевіркою і аудитом перегляду PII).
4. Історична аналітика — тільки з DWH/BI (ніколи з OLTP: захист прод-бази
   і коректність метрик).

### 2.4 PII і зони даних (RODO/GDPR)

- **Зона A (жорстка)**: документи KYC, платіжні токени — окремі таблиці,
  шифрування, доступ 2-3 ролям, кожен перегляд в аудиті.
- **Зона B (звичайна PII)**: імена, контакти — доступ за роллю.
- **Зона C (аналітична)**: у DWH PII псевдонімізується (user_id-хеш);
  аналітики не бачать імен/контактів.
- Right to erasure: анонімізація A/B зі збереженням транзакційних фактів
  (легальна підстава — бухгалтерський облік), каскад у DWH.
- Резидентність: ЄС (eu-central-1), DPA з провайдерами.

### 2.5 Масштабування даних

- Ключі шардування закладаємо зараз: `market` (країна+місто) у Listings/
  Booking; `user_id` в Identity. До шардів — партиціювання Postgres.
- Валюта: кожна грошова величина = amount+currency (ADR-0002); курси —
  окремий довідник з датою; ledger веде рахунки у валюті рахунку.
- Мови: контент host-ів зберігається мовою оригіналу + машинний переклад
  на льоту з кешем (модель Airbnb), UI-локалізація — ресурсні файли.
- Нові типи житла/зручностей/правил — записи довідників (конфігурація),
  не міграції коду.
- Нові платіжні методи — адаптер у Payments за спільним інтерфейсом
  (charge/hold/capture/refund/payout).
