# 06 — Управління конфліктами (Resolution Center)

Модель — синтез практик Airbnb Resolution Center / AirCover,
Booking.com partner disputes і платіжних мереж (chargeback-процес):
**спершу сторони, потім докази, потім людина, рішення платформи —
остаточне і виконуване грошима через Ledger.**

## 1. Типи конфліктів

| Тип | Приклади | Специфіка |
|---|---|---|
| Guest → Host | не відповідає опису, брудно, не зміг заселитись, host скасував | найчастіші; вікно подачі — до 72 год після виявлення, для заселення — 24 год |
| Host → Guest | пошкодження майна, порушення правил, понаднормові гості | claim-вікно: 72 год після check-out, до виплати гостю наступного бронювання |
| Guest ↔ Platform | незгода з поверненням, з блокуванням, зі стягненням | розглядає інша роль, ніж та, що ухвалила первинне рішення |
| Host ↔ Platform | незгода зі штрафом, санкцією, делістингом, рішенням по суперечці | те саме; апеляція одна |

## 2. Чотири рівні вирішення

### Рівень 1 — Пряме врегулювання (auto-negotiation)
- Ініціатор створює кейс: типізована причина + бажаний результат
  (повернення X, компенсація Y, інше).
- Система відкриває структурований діалог: інша сторона за **24 год**
  (таймер, нагадування на 12-й год) відповідає: погодитись / зустрічна
  пропозиція / відхилити.
- До 2 раундів пропозицій. Згода → авто-виконання через Ledger, кейс
  `resolved_by_parties`. **Мета: ≥50% кейсів закриваються тут** (бенчмарк
  Airbnb Resolution Center).

### Рівень 2 — Автоматизований збір доказів
- Тригер: відхилення/тайм-аут рівня 1.
- Система вимагає докази за **чек-листом типу кейсу** (пошкодження: фото
  до/після, чеки, інвентарний список; невідповідність: фото при заселенні,
  таймстемпи): у кожної сторони **48 год**.
- Автоматично підтягуються: листування Messaging, статуси бронювання,
  історія проблем host-а/guest-а, чек-листи прибирання з Operations
  (синергія: фотозвіт прибиральника — доказ стану до заїзду!).
- Дедлайн пропущено → кейс йде далі з наявними доказами (позначка
  «сторона не надала»).

### Рівень 3 — Рішення адміністратора (Dispute Resolver)
- SLA: рішення за **72 год** з моменту передачі.
- Resolver працює по **матриці рішень** (тип кейсу × докази → діапазон
  компенсації); відхилення від матриці — з письмовим обґрунтуванням.
- Рішення: повне/часткове повернення, компенсація host-у з депозитної
  відповідальності guest-а, штраф, ризик-сигнал, без задоволення.
- Виконання — автоматичне: проводки Ledger, оновлення статусів.

### Рівень 4 — Апеляція і остаточне рішення платформи
- Одна апеляція на кейс, вікно **7 днів**, лише за нових доказів або
  процедурної помилки.
- Розглядає senior-роль (не той самий resolver). SLA 5 робочих днів.
- Рішення остаточне. Далі — лише зовнішні механізми (чарджбек, суд);
  чарджбек після програної апеляції → автоматичний ризик-сигнал guest-у.

## 3. Статусна машина Dispute

```
opened → negotiation (L1, 24h×2) → evidence (L2, 48h)
       → under_review (L3, 72h) → decided → appeal_window (7d)
       → appealed (L4, 5 днів) → final
Термінальні: resolved_by_parties, decided_final, withdrawn, expired
```

Супутні ефекти будь-якого відкритого dispute:
- пов'язана виплата host-у → `frozen`;
- перехід бронювання `checked_in → completed → closed` блокується;
- сторони бачать таймери і наступний крок (прозорість знижує ескалації).

## 4. SLA-зведення і автоматичні нагадування

| Етап | Таймер | Нагадування | Прострочення |
|---|---|---|---|
| Відповідь сторони на L1 | 24 год | 12 год | авто-перехід на L2 |
| Зустрічний раунд L1 | 24 год | 12 год | авто-перехід на L2 |
| Докази L2 | 48 год | 24 год | розгляд без доказів сторони |
| Рішення L3 | 72 год | 48 год (resolver-у) | ескалація lead-у, кейс у SLA-звіті |
| Апеляція (подача) | 7 днів | 5-й день | рішення стає final |
| Розгляд L4 | 5 роб. днів | 3-й день | ескалація Head of Ops |
| P1 «не можу заселитись» | окремий трек: host 2 год → support негайно | 1 год | авто-рішення на користь guest-а |

## 5. Гроші суперечок

- Джерела виконання рішень: escrow бронювання (до виплати), майбутні
  виплати host-а (утримання), **депозитна відповідальність guest-а**
  (переавторизація картки; фізичний депозит — рішення на V1),
  фонд гарантій платформи (аналог AirCover — параметри після
  затвердження монетизації).
- Всі рухи — проводки Ledger з посиланням на кейс: аудит повний.

## 6. Анти-зловживання

- Ліміт частоти кейсів (guest, що судиться після кожного проживання →
  ризик-профіль, підвищений стандарт доказів).
- Симетрично для host-ів (систематичні claim-и на депозити → T&S-рев'ю).
- Фальшиві докази = підстава для `suspended` + анулювання рішень.
