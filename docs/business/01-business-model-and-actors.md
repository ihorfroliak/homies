# 01 — Бізнес-модель і учасники системи

## 1. Бізнес-модель платформи

### 1.1 Сутність бізнесу

Homies — **двосторонній керований маркетплейс** (managed marketplace)
короткострокової та гнучкої оренди житла. Платформа не володіє житлом;
вона створює і монетизує **довіру та ліквідність**:

- для Guest — гарантія, що об'єкт існує, відповідає опису, оплата захищена,
  проблема буде вирішена;
- для Host — гарантія платоспроможного гостя, захист від збитків,
  інструменти управління (календар, ціни, аналітика) і передбачувані виплати;
- для платформи — комісія з транзакцій + додаткові потоки (деталі
  монетизації — після затвердження, крок 13; тут фіксуємо лише **структуру**
  потоків: комісія guest-side і host-side, SaaS-інструменти для професійних
  host-ів, просування оголошень, суміжні сервіси — прибирання/обслуговування,
  синергія з HeyHomie).

### 1.2 Чому «керований» маркетплейс

Чистий класифайд (OLX-модель) не захищає транзакцію і не масштабує довіру.
Airbnb/Booking виграли, бо взяли на себе: оплату (escrow-механіка),
верифікацію, страхові гарантії, вирішення суперечок, стандарти якості.
Homies проєктується одразу так само: **платформа — сторона кожної
транзакції**, а не дошка оголошень.

### 1.3 Ефекти масштабу, закладені в модель

| Ефект | Механізм у Homies |
|---|---|
| Мережевий ефект (локальний) | Ліквідність міряється на рівні «місто × сегмент»; запуск нішами (місто за містом), supply-first |
| Економія на довірі | Рейтинги, верифікація, гарантії — цінність росте з кількістю транзакцій |
| Дані як актив | Кожна транзакція покращує ціноутворення, пошук, антифрод, прогноз попиту |
| Операційний важіль | Автоматизація процесів → адмін-команда росте повільніше за GMV |

### 1.4 Три системи платформи

| Система | Продукт | Основна мета |
|---|---|---|
| **Guest Platform** | веб + мобільний застосунок | знайти → забронювати → прожити → повернутись |
| **Host Platform** | веб-кабінет + мобільний застосунок | опублікувати → керувати → заробити → зростати |
| **Admin Operations Center** | внутрішній веб-продукт | бачити все в реальному часі → керувати винятками → тримати якість і ризики |

Системи незалежні у флоу та UI, але працюють над спільними доменними
сутностями і спілкуються подіями (ADR-0004).

## 2. Учасники системи та ролі

### 2.1 Зовнішні актори

| Актор | Підтипи | Опис і особливості |
|---|---|---|
| **Guest** | B2C-фізособа; **B2B-акаунт** (компанія бронює для працівників — релокації, відрядження) | B2B має: центральний акаунт-платник, під-профілі мандрівників, інвойси на компанію, ліміти бронювань |
| **Host** | (a) приватний власник 1–2 об'єктів; (b) **професійний host / property manager** (десятки-сотні об'єктів); (c) юрособа | Різні вимоги KYC, різні інструменти (bulk-операції, API/channel manager для типу b), різний податковий трек |
| **Co-Host** | помічник host-а | Делеговані права на підмножину об'єктів: календар, повідомлення, ціни — але не виплати і не юридичні дані. Необхідний: у професійних host-ів завжди працює команда |
| **Field Worker** | прибиральник, майстер, інспектор | Виконує задачі Operations (прибирання після виїзду, ремонт, фотоінспекція). Пряма синергія з HeyHomie |
| **Партнери** | платіжний провайдер, страховий партнер, канал-менеджери (OTA-синхронізація), картографія, KYC-провайдер | Інтегруються через адаптери; не мають прямого доступу до домену |

### 2.2 Внутрішні актори (Admin Operations Center)

| Роль | Зона відповідальності | Ключове обмеження |
|---|---|---|
| **Support Agent (L1)** | звернення guest/host, чати підтримки, прості кейси | не може рухати гроші, не бачить повних PII-платіжних даних |
| **Support Lead (L2)** | ескалації, компенсації до ліміту, SLA-контроль | компенсації в межах ліміту ролі |
| **Moderator** | модерація оголошень, фото, відгуків, профілів | не бачить фінансів |
| **Trust & Safety Officer** | KYC/AML-рев'ю, фрод-кейси, блокування, ризик-правила | будь-яке блокування — з фіксацією підстави |
| **Dispute Resolver** | суперечки 3-го рівня, рішення по claim-ах | рішення в межах політик; понад ліміт — Finance Ops |
| **Finance Ops** | виплати, повернення, звірки (reconciliation), інвойси | подвійне підтвердження на ручні рухи грошей |
| **City/Market Manager** | supply-акквізиція в місті, якість інвентаря, локальні промо | комерційна роль, працює з дашбордами |
| **Platform Admin** | конфігурація платформи, ролі, права, довідники | не операційна роль |
| **Super Admin** | керування адміністраторами, аварійні дії | усе під аудитом, 2FA обов'язково |
| **System (автоматика)** | окремий актор! всі автоматичні дії виконуються від імені `system` з посиланням на правило | кожна авто-дія — подія в аудиті |

### 2.3 Принципи системи ролей і дозволів (RBAC + ABAC)

1. **RBAC як основа**: роль = набір дозволів (permission). Дозволи атомарні
   (`booking.refund.create`, `listing.moderate`, `user.block`).
2. **ABAC-обмеження зверху**: атрибути звужують дію дозволу — ліміт суми
   компенсації, тільки свій ринок (місто/країна) для City Manager, тільки
   делеговані об'єкти для Co-Host.
3. **Розділення обов'язків (SoD)**: той, хто ухвалив рішення по суперечці,
   не виконує виплату; ручна виплата понад ліміт — два підтвердження
   (maker-checker).
4. **Кожна дія — запис аудиту**: хто, коли, що, до/після, на якій підставі
   (правило або тікет). Аудит незмінюваний (append-only).
5. **Ролі — дані, не код**: нові ролі й ліміти створюються конфігурацією.

### 2.4 Матриця «актор → системи»

| Актор | Guest Platform | Host Platform | Admin Center |
|---|---|---|---|
| Guest | ✅ повний | — | — |
| Host | ✅ (може бути і гостем — один акаунт, дві ролі) | ✅ повний | — |
| Co-Host | — | ✅ делеговані об'єкти | — |
| Field Worker | — | ✅ мобільний застосунок задач | — |
| Support/Moderator/T&S/Finance/Admin | імперсонація read-only для діагностики (під аудитом) | так само | ✅ за роллю |

**Рішення: єдиний акаунт — кілька ролей.** Користувач Homies має один
обліковий запис; «Guest» і «Host» — це профілі-ролі поверх нього
(як Airbnb). Причини: спільна репутація та KYC, простіший вхід host-а
в guest-сценарії, одна історія платіжних методів. Це узгоджено з
рішенням про спільну ідентичність із HeyHomie (charter, seam
`ClientAccount`).
