# EA Review — Part 2/3: Red Team (100 причин провалу) + Benchmark

## Red Team: 100 причин, чому Homies провалиться в продакшні

Формат: причина → мітигація. Згруповано. Нумерація наскрізна.

### A. Ринок і cold start (1–14)
1. Порожня видача в день запуску → сідінг 100+ об'єктів прямими продажами до публічного запуску.
2. Host-и не приходять без guest-ів, guest-и без host-ів → supply-first + гарантії першим host-ам (0% комісії, мін. дохід).
3. CAC у Google Ads проти Booking незйомний → SEO+B2B-релокації як основні канали, Ads лише ретаргет.
4. Мультихомінг host-ів → подвійні бронювання → iCal-sync у V1, штраф не карає, а вбиває — компенсаційний флоу переселення.
5. Немає диференціації від Airbnb → зафіксувати ціннісну відмінність (гнучкі терміни + інтегрований клінінг + host-економіка) до витрат на маркетинг.
6. Ліквідність розмазана по містах → жорсткий фокус: одне місто до цільової liquidity-метрики.
7. Сегмент помісячної оренди має інших конкурентів (Homfi-класу) з іншою механікою (договори najmu) → окремий продуктовий трек, не «той самий букінг довше».
8. Сезонність попиту ламає юніт-економіку → B2B-база як стабілізатор.
9. Host-и йдуть після 1-2 поганих гостей → депозити+гарантії з V1, не V2.
10. Guest-и не довіряють новому бренду гроші → платіжний партнер з ім'ям (Stripe/PayU) на чекауті, гарантія повернення прописана золотими літерами.
11. Реферали відсутні → host-referral програма (найдешевший supply).
12. Повторні бронювання без механіки повернення → тригерні кампанії V1.
13. PM-и не можуть зайти (нема bulk/API) → мінімальний PM-пакет V1.
14. Локальні гравці демпінгують комісією → модель доходу не тільки з комісії (SaaS-інструменти, сервіси).

### B. Регуляторика і право (15–26)
15. Прийом коштів без платіжної ліцензії = нелегальна діяльність → Stripe Connect/Mangopay/Adyen, ADR до першого рядка Payments.
16. Короткострокова оренда регулюється (реєстрація об'єктів, ліміти днів у містах ЄС) → реєстр вимог по ринку, поле «номер реєстрації» в Listing з дня 1.
17. EU STR Regulation (обмін даними з владою) → архітектурно передбачити expor-канал даних про бронювання.
18. DAC7-звітність доходів host-ів → збір податкових ідентифікаторів у KYC, річний звіт.
19. DSA-обов'язки платформи (notice-and-action, прозорість модерації) → процес скарг на контент з відповідями.
20. GDPR: consent-трекінг, retention schedule, DPIA для скорингу → до запуску, не після.
21. Міський туристичний податок → конфігуровані податки ринку в Pricing.
22. VAT на комісію та інвойсинг юрособам → податкова логіка в Ledger/Payments з бухгалтером.
23. Умови користування без юриста → travel/marketplace-юрист, T&C по ринку.
24. Дискримінаційні відмови host-ів (заборонено) → правила контенту+поведінки, моніторинг відмов по патернах.
25. Претензії щодо нещасних випадків у житлі → страховий партнер, межі відповідальності в T&C.
26. LE-запити без процесу → формальний legal request процес.

### C. Гроші (27–38)
27. Refund після виплати → негативні баланси без стягнення → utримання з майбутніх виплат + ліміт експозиції на host-а.
28. Chargeback-хвиля від фрод-кільця → ліміти нових host-ів, затримка виплат перших N бронювань.
29. FX-розриви між оплатою і refund → правило «refund у валюті і сумі оригінального платежу», курсовий ризик платформи хеджується у провайдера.
30. SCA/3DS ламає конверсію → exemption-стратегія, MIT-мандати на доплати.
31. Ledger розходиться з провайдером → щоденна звірка з нульовою толерантністю, блокер-алерт.
32. Подвійна виплата при retry → ідемпотентність payout-ів по ключу, стан у ledger, не в пам'яті.
33. Невиплачені кошти «зависають» (host без валідного IBAN) → dormant-процес, ескалація, юридичні строки.
34. Округлення часток комісій → правила округлення в мінорних одиницях, залишки на технічний рахунок.
35. Купони+знижки дають від'ємну ціну → інваріанти цін у Pricing (мін. ціна > зборів).
36. Часткові скасування (1 ніч із 30) не порахуються → пропорційні правила в політиках з тестами.
37. Виплата на вкрадений IBAN (ATO) → cooling period 48h після зміни реквізитів + MFA + сповіщення.
38. Податкова перекваліфікація моделі (агент vs принципал) → юр-висновок до масштабування.

### D. Trust & Safety (39–50)
39. Фантомні оголошення збирають передоплати → верифікація об'єкта (фото+гео+документ власності для нових host-ів до першої виплати).
40. Кільця host-guest відмивають крадені картки → graph-аналіз зв'язків, затримка виплат новим host-ам.
41. ATO гостьових акаунтів зі збереженими картками → MFA-опція, аномалії входу, ліміти MIT.
42. Фейкові відгуки накачують рейтинг → verified-stay only (є) + graph + текстовий ML.
43. Обхід платформи після першого контакту → маскування (є) + цінність залишатися (гарантії, підтримка) — комунікувати, не лише карати.
44. Скрейпінг цін і контенту ботами → rate limiting, fingerprinting, поведінковий захист.
45. CSAM/незаконний контент у чатах → сканування + процес репортингу (юр. обов'язок).
46. Санкційні особи серед host-ів → screening при KYC і періодично.
47. Внутрішній фрод співробітника з правами → SoD, аудит, алерти на аномальні дії ролей.
48. Дискримінація/харасмент у чатах → модерація повідомлень, скарги, санкції.
49. Небезпечні об'єкти (газ, електрика) → чек-лист безпеки host-а, поле «датчик диму», страховий партнер.
50. Сусіди/шум → «party house»-детектор (Airbnb-практика: ліміт гостей, локальні скарги).

### E. Безпека (51–62)
51. Admin-панель зламана через слабкий пароль → SSO+MFA обов'язково, IP-сигнали, короткі сесії.
52. IDOR в admin/host API → ABAC-тести в CI на кожен ендпоінт.
53. Webhook-и провайдера підроблені → підписи+replay window+allowlist.
54. Секрети в репо/compose → сканер секретів у CI, Secrets Manager, ротація.
55. SQL-ін'єкції/OWASP-базовиe → параметризація (ORM), SAST+DAST у CI, security headers.
56. Витік PII з DWH → псевдонімізація (є) + row-level доступ + аудит запитів.
57. Rate limiting відсутній → ліміти per-IP/user/endpoint з дня 1.
58. JWT-крадіжка → короткий TTL, ротація refresh (є), device binding, revocation list.
59. Залежності з CVE → dependabot/renovate + SCA-скан у CI.
60. Бекапи незашифровані/нетестовані → шифрування + квартальні відновлення.
61. DDoS на пошук → CDN/WAF перед публічними ендпоінтами.
62. Phishing на host-ів «підтвердіть виплату» → комунікаційна політика (ніколи не просимо X), маркування офіційних повідомлень.

### F. Архітектура і масштаб (63–74)
63. CalendarDay-таблиця стає гарячою точкою → інтервальна модель + exclusion constraints, календар як проєкція.
64. Подвійне бронювання при split-brain → транзакційний інваріант лише в Postgres, ніколи в кеші.
65. Меilisearch розсинхронізувався → reindex-процедура, лаг-метрика, чесність календаря перевіряється в Booking завжди.
66. Події губляться (немає outbox) → transactional outbox + at-least-once + ідемпотентні консюмери.
67. Саги застрягають посередині → таймаути, компенсації, дашборд «застряглих» процесів.
68. Модуль читає чужі таблиці «тимчасово» → архітектурні тести (import-linter) у CI.
69. Моноліт деплоїться годину через міграції → zero-downtime міграції (expand-contract) як норма.
70. Ledger повільний на балансах → ES+snapshot-проєкції балансів.
71. Мультивалютність вшита пізно → amount+currency скрізь уже (ADR-0002) — тримати лінію, курс-довідник з дня 1.
72. i18n retrofitting болючий → UI-рядки в ресурсах з дня 1, контент з мовним тегом.
73. Часові зони ламають check-in логіку → всі часи в UTC + timezone об'єкта в Listing, тести на межі дат.
74. Контракти дрейфують від коду → codegen з OpenAPI + contract-тести в CI.

### G. Операції і підтримка (75–84)
75. P1 вночі нема кому взяти → авто-рішення на користь guest-а поза годинами + чесний SLA.
76. Черга модерації росте швидше за модераторів → автомодерація ML з першого місяця, люди на сірій зоні.
77. Суперечки з'їдають маржу часом людей → L1-автопереговори з стимулами до згоди (статистика Airbnb: більшість закривається до людини).
78. Клінінг не встигає між заїздами → мін. розрив (є) + realtime-статус задачі, без «прибрано» — нема нового check-in.
79. Знання в голові засновника → runbook-и як частина DoD кожної фічі.
80. Пікові дні (新год, Євро-2028) кладуть операції → capacity-план черг, freeze нових правил у піки.
81. Провайдер платежів ліг → degraded mode: бронювання в pending, чесний банер, черга на повтор.
82. Масові скасування форс-мажор → bulk-EC процес (Part 1, П3).
83. Відповіді підтримки трьома мовами → AI-переклад з рев'ю + мовний роутинг.
84. CSAT ніхто не міряє → CSAT після кожного тікета, ціль у KPI.

### H. Дані/ML (85–91)
85. Метрики брешуть через різні визначення → метрик-словник (dbt metrics) як єдине джерело.
86. Події ламаються мовчки → schema registry + алерти на обсяг/схему.
87. A/B нема — рішення на смак → мінімальна experimentation-рамка до growth-витрат.
88. Smart Pricing занижує ціни всім → guardrails (min/max host-а), моніторинг revenue-impact, поетапний rollout.
89. Фрод-ML банить чесних → людське рев'ю над порогом, метрика false positive, апеляція.
90. GDPR-видалення не доходить до DWH → каскадна анонімізація з тестом.
91. Рекомендації радять недоступне → фільтр доступності на serving-етапі.

### I. Виконання (92–100)
92. **Соло-розробник вигорає — найімовірніший сценарій провалу** → нещадне скорочення scope (Part 3), тижневі віхи, «задеплоєне > повне».
93. 17 контекстів документації > швидкість розробки → документи слідують за кодом після MVP, не навпаки.
94. Аналіз-параліч (ще один пакет документів замість коду) → правило: жоден новий документ, поки не працює booking happy path.
95. Технології заради резюме (Kafka, EKS) на 0 користувачів → NATS+один VPS/ECS до реальної потреби; charter уже це визнає — дотриматися.
96. Немає зовнішнього фідбеку до «готовності» → 5 інтерв'ю з host-ами (у користувача Є доступ через HeyHomie-клієнтів!) до V1.
97. Бюджет хмари з'їдає runway → cost-ліміти, калькуляція в фінмоделі (Чат 13).
98. Залежність від одного платіжного партнера → абстракція адаптера (є) + план Б у документах.
99. HeyHomie-синергія залишиться слайдом → перший спільний флоу (клінінг після виїзду) в V1 як proof.
100. Проєкт не має визначення «успіху» → явні критерії: портфоліо (демо+кейси до дати X) чи бізнес (N бронювань/міс до дати Y) — різні плани.

## Benchmark проти лідерів

| Платформа | Що в них є, чого нема в Homies | Чому важливо |
|---|---|---|
| **Airbnb** | AirCover (гарантія $-обсягу для обох сторін); Superhost-програма; Experiences; гнучкий пошук («куди завгодно»); верифікація об'єктів; party-detection | довіра як продукт, не функція; retention host-ів |
| **Booking.com** | Genius-лояльність рівнями; guest checkout без акаунта; агресивна CRM-машина; готельний inventory (ширина) | конверсія і частота використання |
| **Vrbo/Expedia** | сімейний сегмент-фокус; пакетні тарифи (переліт+житло) | сегментація замість «для всіх» |
| **Agoda** | цінова агресія в Азії, прайс-дисплей A/B-машина | ціна як алгоритмічний продукт |
| **Uber** | real-time ops-центр як ядро компанії; surge-механіка; миттєвий підбір | Live Monitoring у Homies задекларований — в Uber це серце, вчитися їх playbook-ам інцидентів |
| **Amazon Marketplace** | seller performance-метрики з автоматичними санкціями; A-to-Z Guarantee; FBA (платформа бере операції на себе) | Homies-аналог: «managed hosting» — платформа повністю веде об'єкт (клінінг+ключі+прайсинг) = HeyHomie-синергія, потенційно головна диференціація |
| **Stripe** | ідемпотентність скрізь, версіоновані API, radar-фрод, звіти-як-продукт | інженерна культура грошей — Ledger вже в цьому напрямі |
| **Shopify** | екосистема застосунків/партнерів, API-first | V3-перспектива партнерського API — правильна, але після ядра |

**Головний висновок бенчмарку:** у Homies є одна потенційно унікальна
карта, якої нема в жодного конкурента такого масштабу локально —
**вертикальна інтеграція операцій житла через HeyHomie (клінінг,
обслуговування, інспекції)**. «Managed hosting»-продукт (host віддає
ключі — платформа робить усе) = Amazon FBA для оренди. Рада рекомендує
зробити це стратегічним центром диференціації замість гонитви за
функціональним паритетом з Airbnb.
