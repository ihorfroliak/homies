# D7 — Production Readiness Board: Go/No-Go (evidence-only)

> Рада ухвалює: чи можна запустити Homies із **реальними хостами,
> гостями і грошима завтра**. Правило: докази важать більше за наміри;
> незбудоване = відсутнє; непротестоване = неперевірене; невиконане
> відновлення = недоведене. Дата: 2026-07-05. Оцінюється **commit dd3b88c**.

## 2. Money Protection (кожне євро — чуже)

**Доведено (warfare, живий Postgres):** ledger не дублює і не губить —
200× webhook-шторм → 1 capture; 10× payout → 1 виплата; refund-loop ×5 →
escrow щоцикл у нуль; крах БД → ACID, recon=0. Інваріанти escrow≥0,
append-only, сума=0 тримаються.

**Але гроші МОЖУТЬ зникнути за таких умов (доказові діри):**
1. **Крах диска БД** → безповоротна втрата всіх грошей-фактів:
   **PITR-бекапів немає** (ACID рятує від часткового запису, не від
   втрати тому). Це **found, not assumed**.
2. **Реальний провайдер** не підключено → нема звірки ledger ↔ реальний
   Stripe Balance; розсинхрон із реальністю **не може бути виявлений**,
   бо реальності нема.
3. **Refund після виплати** → 409 (свідомо), але clawback-флоу відсутній
   → у проді це негативний баланс host-а без стягнення.
4. **Chargeback** — обробки немає взагалі.

Вердикт §2: облік **не дає грошам зникнути всередині системи**
(доведено), але **периметр збереження грошей (бекапи, реальний
провайдер, chargeback) відсутній** → у проді гроші втратити можна.

## 3. Operational Readiness (100 host / 500 listing / 5000 guest / 1000 booking завтра)

**Ні.** Доказ: 0 виконаних runbook-ів, 0 on-call, 0 моніторингу/алертів,
1 людина, ручний `create_admin`-скрипт як єдина операційна процедура.
Ghost-booking-черга неоплачених накопичується без сигналу (D6: 34
requires_payment зависли за прогон). При 1000 бронювань/день система
операційно **сліпа**: інцидент виявляється лише скаргою клієнта.
Deployment — `docker compose up`, rollback-процедури нема, CI жодного
разу не проходив.

## 4. Security Governance

| Контроль | Стан (доказ) |
|---|---|
| Identity lifecycle | базовий; нема suspend/ban-флоу в коді |
| Session mgmt | refresh-ротація є (доведено); нема device-list/remote-logout |
| Credential rotation | **немає**; JWT-секрет — статичне dev-значення в конфізі |
| MFA | **0 у коді** |
| Admin privilege separation | 1 роль admin, повний доступ; **нема maker-checker** |
| Least privilege | admin бачить усе; імперсонація/PII-аудит не реалізовані |
| Secret management | **dev-секрети в config.py**; Secrets Manager не підключено |
| Audit immutability | ORM-guard доведено; **DB-рівень REVOKE відсутній** → admin з доступом до БД може мовчки правити ledger |

**Ключовий висновок §4:** внутрішній адміністратор з доступом до
Postgres **може мовчки зіпсувати фінансову цілісність** (ORM-guard не діє
на прямий SQL; DB-REVOKE немає). Це доведена діра governance.

## 5. Disaster Recovery

| Параметр | Стан | Доказ |
|---|---|---|
| RPO | **невизначений** | бекапів немає → RPO = «від останнього дампу», якого нема |
| RTO | ~15 c для рестарту процесу | D6 (єдиний доведений сценарій) |
| Backup strategy | **відсутня** | нема pg_dump/PITR/snapshot |
| Restore validation | **ніколи не виконувалась** | → DR вважається **недоведеним** |
| Region outage / failover | **немає** | одна БД, один хост = SPOF |
| Key rotation | **немає** | статичні секрети |

DR-статус: **UNPROVEN** (за правилом ради — невиконане відновлення не
зараховується). Доведено лише ACID при рестарті одного контейнера.

## 11–13. Обов'язкові дії

**P0 (до будь-яких реальних грошей):** B1 реальний Stripe Connect за
наявним `PaymentProvider`-швом + webhook-підпис; B2 PITR-бекапи +
**виконаний** тест відновлення; B3 мінімальний комплаєнс-пакет (юрист:
T&C, KYC через Stripe, GDPR consent/erasure); B5 DB-REVOKE на
ledger-таблицях; B6 Alembic + перша міграція (винести exclusion
constraint); B4 auto-void + rate-limit.

**P1 (перші 30 днів проду):** observability (метрики/логи/алерти,
синтетичний happy-path), MFA адмінів + secrets-mgmt, chargeback/clawback-
флоу, щоденна авто-звірка ledger↔Stripe, runbook-и топ-5 відмов
(виконані в staging), CI на реальному remote.

**P2 (90 днів):** disputes-модуль, verified-listing, fraud-скоринг,
maker-checker, DAC7-звіт, VAT/KSeF-інвойси, DR-навчання (region-outage).

## 18. Executive Go / No-Go

**Головне питання:** якщо ти особисто відповідаєш за кожне євро, кожного
host-а і кожного гостя — чи дозволиш реальний прод-трафік сьогодні?

# ⛔ NO-GO

**Доказ рішення (не думка):** через систему не пройшло жодного реального
євро (провайдер симуляція, B1); втрата диска БД = безповоротна втрата
грошей-фактів (бекапів немає, B2); жоден комплаєнс-домен не Ready (B3);
внутрішній admin може мовчки правити ledger (B5); система сліпа до
інцидентів (B7). Будь-якого одного з B1–B3 достатньо для NO-GO;
присутні всі.

**Що ядро таки заслужило (чесно):** фінансово-облікова і конкурентна
логіка **warfare-доведена** (D6) — це реальний актив, не папір. Рішення
NO-GO стосується **продакшн-обгортки**, не якості ядра.

### Дозволений обмежений режим (не «реальні гроші»)
**GO-WITH-RESTRICTIONS** можливий лише для: **managed-operator пілот у
Stripe test-mode** — реальні host-и онбордяться, реальні клінінг-операції
HeyHomie виконуються, але **бронювання і гроші — тільки sandbox**, ручні
операції засновника, без публічного трафіку. Це дає юніт-економіку
(CM2) без ризику реальних коштів, поки закриваються P0. Узгоджено зі
стратегією (пілот 10 об'єктів, docs/strategy/07).
