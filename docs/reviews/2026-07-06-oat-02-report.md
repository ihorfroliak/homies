## 3. Regression — money core intact (warfare, live Postgres)

All D6 verdicts PASS after OAT-02: 60 concurrent→1 booking, **200 webhook→1
capture**, 10 payout→1, ledger-integrity PASS, reconciliation ok=True,
double_capture=[].

**Bug caught + fixed this cycle:** re-running warfare surfaced a latent
**double-capture race** in `process_intent_succeeded` (two concurrent webhook
deliveries could both pass the `status==succeeded` guard before either
committed). Fixed at root cause with a `SELECT ... FOR UPDATE` row lock on the
payment — deterministic single capture under 200 concurrent deliveries.
(D6 had passed by timing luck; the per-cycle warfare re-run is what exposed
it — the value of Build→Verify.)
